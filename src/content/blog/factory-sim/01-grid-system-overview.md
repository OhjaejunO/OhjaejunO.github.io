## 1. 들어가며 — 자동차 공장에서 UE5까지

기아 오토랜드 화성공장에서 약 1년 3개월 일했습니다. 컨베이어가 도는 라인 옆에서, 설비가 멈추면 어디서 뭐가 막혔는지 추적하던 시간이었어요. 그 경험 덕에 공장이라는 시스템이 어떻게 흐르는지에는 감이 있는 편이지만, 정작 그걸 다른 사람에게 보여줄 도구는 갖고 있지 않았습니다.

지금은 부트캠프에서 UE5를 배우고 있고, 커리어 방향은 **디지털 트윈 / 스마트팩토리** 쪽으로 잡았습니다. 현실 공정을 가상 환경에 옮기고, 거기서 시뮬레이션·예측·시각화를 돌리는 일이요. 자동차 공장 출신이 게임 엔진을 잡으면 이 분야가 의외로 자연스럽게 이어집니다.

지금 진행 중인 프로젝트는 두 갈래입니다.

- **Wanted_Factory** (팀 프로젝트) — 부트캠프 트랙에서 만드는 공장 시뮬레이션 게임. UE5 + C++.
- **DigitalTwinFactory** (개인 프로젝트) — MQTT + Pixel Streaming으로 실제 공정 라인을 가상화하는 사이드 트랙.

이번 시리즈는 그중 **Wanted_Factory** 작업기입니다. 그리고 이 글이 첫 화이기도 하고, 더 큰 메타 시리즈의 출발선이기도 합니다.

## 2. 왜 공장 시뮬레이션 게임인가

장르부터 잠깐 정리하고 갈게요.

**자동화 / 공장 시뮬레이션 장르**의 게임은 대체로 이런 구조를 공유합니다.

- 빈 땅 위에 **머신을 셀 단위로 배치**한다.
- 머신은 **자원을 입력받아 가공해 다른 자원으로 출력**한다.
- 머신끼리는 **컨베이어 / 파이프 / 암 같은 운반 수단**으로 연결된다.
- 시간이 지나면 라인이 **자율적으로 돌아가야** 한다.

여기까지만 봐도 이미 공장 그 자체입니다. 셀, 풋프린트, I/O 포트, 사이클 타임, 처리량 — 게임 안에서 쓰는 용어와 현장에서 쓰는 용어가 거의 일치해요. 그래서 이 장르의 게임을 직접 만들어보는 건, 디지털 트윈을 배우기에 꽤 좋은 학습 도구가 됩니다.

특히 제가 흥미를 느끼는 지점은 이쪽입니다.

- **게임 = 실제 공장 시스템의 미니어처**라는 시점.
- 게임 안에서 잘 굴러가는 아키텍처가, 그대로 실제 디지털 트윈에도 옮길 수 있는 토대가 된다는 가설.
- 자동차 공장에서 본 라인 구조(투입 → 가공 → 검사 → 출고)와 게임 시스템 사이의 매핑.

이 시리즈가 단순 UE5 튜토리얼이 아닌 이유가 여기에 있습니다. **게임을 만들면서, 디지털 트윈의 뼈대를 같이 설계하고 있는** 셈이에요.

## 3. Phase 1 — 그리드 시스템이란

본격적으로 첫 번째 페이즈로 들어갑니다. 공장 시뮬레이션 게임의 가장 밑바닥에는 **그리드 시스템**이 있습니다. 머신을 어디에 어떻게 놓을지 결정하는 토대요.

### 셀 기반 배치를 선택한 이유

머신 배치 방식은 크게 두 가지로 나뉩니다.

| 방식 | 장점 | 단점 |
|---|---|---|
| **자유 배치** (free placement) | 자연스러운 레이아웃, 미적 자유도 | 충돌 검증·운반 연결이 지옥. 자원 이동 경로 계산을 일반화하기 어려움 |
| **셀 기반** (grid placement) | 충돌·연결이 자명, 데이터 구조가 깔끔 | 시각적 단조로움, 셀 크기에 종속 |

공장 시뮬레이션 장르가 셀 기반을 거의 표준으로 채택하는 이유는 단순합니다. **격자 위에서는 인접·점유·경로 같은 개념이 그냥 `FIntPoint` 산수로 풀립니다.** 자유 배치로 시작했다가 컨베이어/연결 단계에서 다시 격자로 회귀하는 비용이 더 크다고 판단했습니다.

### Phase 1에서 만들 것

이번 페이즈에서 굴러가야 하는 최소 기능은 다음 4가지입니다.

1. **베이스 그리드 표시** — 빌드 모드에 들어가면 바닥에 격자가 깔린다.
2. **호버 미리보기** — 마우스가 가리키는 셀에, 배치할 머신의 풋프린트만큼 **녹색(가능)** 또는 **빨간색(불가)** 표시가 따라온다.
3. **클릭 배치** — 좌클릭 한 번에 머신이 실제로 그 셀에 놓인다. 점유된 셀, 그리드 바깥, 다른 머신과 겹치는 경우는 거부한다.
4. **빌드 모드 토글** — B키로 빌드/플레이 모드를 오간다. 빌드 모드가 아닐 때는 그리드도 호버도 보이지 않는다.

간단해 보이지만 실제로 만들어보면 잡아야 할 잔불이 꽤 많습니다. 그 잔불들을 다섯 편에 걸쳐 풀어볼 예정이에요.

## 4. 아키텍처 한눈에 보기

코드는 다음 글부터 본격적으로 들어가지만, 큰 그림만 먼저 잡고 가겠습니다. Phase 1 그리드 시스템은 **3축**으로 책임을 나눴습니다.

```
┌─────────────────────────────────────────────────────────────┐
│                         입력 레이어                          │
│                AOJJ_BuildController (C++ Actor)              │
│  · 마우스 호버 추적 (커서 → 그리드 좌표)                     │
│  · 좌클릭 → 배치 요청                                        │
│  · B키 → 빌드 모드 토글                                      │
└──────────────────────────┬──────────────────────────────────┘
                           │ TryPlaceMachine(machine, origin)
                           │ UpdateHoverPreview(machine, origin)
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                         데이터 레이어                        │
│                     AOJJ_Grid (C++ Actor)                    │
│  · TMap<FIntPoint, Machine*>  셀 ↔ 머신 양방향 매핑          │
│  · CanPlaceMachine / TryPlaceMachine  트랜잭션 검증·배치     │
│  · WorldToGrid / GridToWorld  좌표 변환                      │
└──────────────────────────┬──────────────────────────────────┘
                           │ 시각화 명령
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                         시각화 레이어                        │
│         GridFloorMesh + ValidHoverISM + InvalidHoverISM      │
│  · M_OJJ_GridFloor 머티리얼 (베이스 격자)                    │
│  · MI 호버용 2종 (녹색 / 빨강 인스턴스드 스태틱 메시)        │
└─────────────────────────────────────────────────────────────┘
```

핵심 원칙은 **책임 분리**입니다.

- `AOJJ_BuildController`는 **입력만** 안다. 어디에 놓을 수 있는지는 모른다.
- `AOJJ_Grid`는 **데이터 상태만** 안다. 마우스가 어디 있는지는 모른다.
- 시각화 컴포넌트는 **그리는 것만** 안다. 누가 그리라고 했는지는 모른다.

이 분리가 잘 안 되면 5편에서 다룰 **숨은 버그들**이 줄줄이 터집니다. 예를 들어 호버 미리보기 액터가 진짜 점유 상태를 건드리면, 마우스만 갖다 댔는데 셀이 점유되는 일이 생겨요. 실제로 한 번 겪었습니다.

## 5. Phase 1 시리즈 미리보기

다섯 편을 어떻게 굴릴 계획인지 미리 풀어둡니다.

**2편 — 그리드 데이터 구조와 트랜잭션 안전성**
`TMap<FIntPoint, Machine*>`만으로는 부족합니다. "이 머신이 점유한 셀은 어디인가"를 역방향으로도 들고 있어야 제거·이동이 O(1)에 풀려요. `TryPlaceMachine`을 트랜잭션으로 묶어, 중간에 실패해도 ghost occupancy가 남지 않게 하는 패턴을 다룹니다.

**3편 — 머티리얼로 그리드 시각화하기**
베이스 격자 한 장을 머티리얼로 그립니다. `Frac` + `Min` 노드로 격자 선을 만들고, 반투명 + 살짝 발광. 호버용 머티리얼은 인스턴스로 두 벌 (녹색 / 빨강) — 머신 풋프린트만큼 셀을 칠해 보여주는 용도입니다.

**4편 — 마우스 입력과 빌드 컨트롤러**
`GetHitResultUnderCursorByChannel`로 마우스 위치를 월드 좌표로 받고, `WorldToGrid`로 셀 좌표로 환산. Tick마다 호버를 갱신하고, 좌클릭에서 `TryPlaceMachine`을 호출하는 흐름. B키 토글로 빌드/플레이 상태 머신 구성.

**5편 — Codex Adversarial Review로 잡은 silent bug들**
Phase 1을 만들면서 AI 도구(Codex)로 adversarial review를 돌렸고, 그 과정에서 **숨은 버그 10개**를 잡았습니다. Ghost occupancy, stale weak pointer, multi-grid contamination, viz/placement coupling 같은 — 컴파일은 통과하는데 게임은 망가지는 부류의 버그들이요. AI 도구를 코드 리뷰어로 쓰는 워크플로우 자체에 대한 글입니다.
